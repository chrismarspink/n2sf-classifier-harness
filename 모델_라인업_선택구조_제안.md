# 등급분류 모델 라인업 & 용도별 선택 구조 제안

> 일자: 2026-07-02 · 목적: 크기별(소형/base/라지) 모델을 **같은 증류 파이프라인**으로 만들어
> **용도(디바이스·지연·정확도)에 따라 선택**하는 구조 설계 + 후보 모델 제안 + 평가 계획

> ⚠️ **§1~8은 라인업 실행 "전" 제안(예측)**입니다. **라인업이 완료되어 실측이 나왔고, 일부 예측이 뒤집혔습니다**
> (XLM-R-large > KLUE-large, 앙상블은 손해). **최신·확정 결과는 아래 [§9 실측 결과](#9-실측-결과-라인업-완료)** 를 보세요.

## 1. 현재 정리 (크기 티어)

- **현재 모델 = `mDeBERTa-v3-base`** — BERT 계열 **base 티어**(트랜스포머 백본 ~86M, 다국어 어휘 임베딩 포함 총 **279M / 558MB**).
- "소형/small"이라 부른 건 **9B LLM 대비 상대 표현**. 계열 내에선 **base**가 정확.
- 성능(held-out): **0.80**(기밀 누락 0), 상한 ~0.78~0.80 → **학생 용량이 병목**.

| 티어 | 파라미터(총) | 크기 | 우리 위치 |
|---|---|---|---|
| 소형(small) | ~14~130M | ~130~300MB | — |
| **base(현재)** | **279M** | **558MB** | ← mDeBERTa-v3-base, F1 0.80 |
| 라지(large) | ~340~560M | ~1.3~2.2GB | 목표 F1 ~0.85 |

## 2. 성능 향상용 한국어/다국어 후보 모델

> 분류 헤드를 붙여 **GPT-4o/o4-mini 교사로 증류**할 인코더 후보. (임베딩계는 exemplar로도 활용 가능)

### 2.1 한국어 특화 (국내 문서 정확도 유리)

| 모델 | 파라미터 | 특징 | 비고 |
|---|---|---|---|
| **KLUE-RoBERTa-large** | ~337M | KLUE 벤치 한국어 최상급, **라지인데 XLM-R-large보다 가벼움** | **정확도 티어 1순위 추천** |
| KLUE-RoBERTa-base | ~110M | 한국어 base 표준 | base 대안 |
| KoELECTRA-base v3 | ~110M | 한국어 ELECTRA, 효율↑ | base 대안 |
| KoELECTRA-small | ~14M | **초경량**, 엣지/실시간 | 소형 티어 후보 |
| KcELECTRA / KPF-BERT | ~110M | 구어체/뉴스 등 도메인 특화 | 문서 특성 맞으면 |

### 2.2 다국어 (혼재 문서·확장성)

| 모델 | 파라미터 | 특징 | 비고 |
|---|---|---|---|
| **XLM-RoBERTa-large** | ~550M | 다국어 라지 표준 | 라지 다국어 후보 |
| mDeBERTa-v3-base (현재) | 279M | 다국어 base, disentangled attention | 현 balanced |
| XLM-RoBERTa-base | ~270M | 다국어 base | base 대안 |
| multilingual-e5 (small/base/large) | 118/278/560M | 의미유사 SOTA급 임베딩 | exemplar/증류 |
| mMiniLM-L12 | ~118M | 경량 다국어 | 소형 티어 후보 |

> **핵심 추천**: 한국어 위주면 **정확도 티어 = KLUE-RoBERTa-large(337M)** — XLM-R-large(550M)보다 가벼우면서
> 한국어 성능이 좋아 **온디바이스 부담과 정확도의 균형**이 낫다.

## 3. 제안: 3-티어 모델 라인업 (용도별 선택)

**같은 교사(GPT-4o/o4-mini) soft-label 증류**로 크기별 학생 3종을 만들고, 용도에 따라 배포·선택한다.

| 라인업 | 모델(후보) | 총크기 | 예상 F1 | 지연(CPU, 뉴럴 경로) | 용도 |
|---|---|---|---|---|---|
| **Fast (소형)** | KoELECTRA-small / mMiniLM | ~130~300MB | ~0.72~0.78 | ~150~300ms | 저사양·엣지, 고처리량, 실시간 |
| **Balanced (base)** | **mDeBERTa-n2sf (현재)** | 558MB | **0.80** | ~0.6s | 일반 노트북·표준 업무 PC |
| **Accurate (라지)** | KLUE-RoBERTa-large-n2sf | ~1.3GB | ~0.85(기대) | ~1.5~3s | 고사양·서버측, 정확도 최우선 |

- 세 모델 **모두 기밀 누락 0(C재현율 1.0)** 를 설정으로 보장(안전은 크기 무관 필수).
- **양자화(int8/ONNX)** 로 각 티어 크기·지연 ~절반 추가 절감 가능.

## 4. 선택 구조 (2가지)

### 4.1 정적 선택 — 디바이스 프로필 (권장 기본)

배포 시 **디바이스/요구사항 프로필**로 모델 하나를 선택해 탑재.

```
프로필 → 모델 선택
  low-spec / 엣지 / 실시간   → Fast(소형)
  표준 PC / 일반 업무        → Balanced(base, 현재)
  서버 / 정확도 최우선       → Accurate(라지)
```
- 하네스의 `NEURAL_BACKENDS`에 3 모델을 등록(이미 확장 구조 보유) → `--model`로 선택.
- 단순·예측가능. 각 티어는 독립 산출물(모델 폴더 + weights.json).

### 4.2 동적 캐스케이드 — 어려운 문서만 승격 (확장 옵션)

한 파이프라인에서 **쉬운 건 빠르게, 애매한 건 정확하게**:

```
문서
 → T1 정규식(즉시 확정 다수, ~3ms) ──[확정]──▶ 등급
 → T2 Fast 소형 모델 (신뢰도 높음?) ──[확정]──▶ 등급
 → T3 Accurate 라지 모델 / (옵션) LLM ────────▶ 등급   ← 소수 애매 문서만
```
- **평균 지연은 소형 수준, 어려운 소수만 라지/LLM** → 속도·정확도 동시 확보.
- "확장성(필요시 LLM)" 포인트와 자연스럽게 연결(온프레미스 유지, 극소수만 외부 옵션).
- 신뢰도 임계(예: 뉴럴 confidence < τ)로 승격 결정.

## 5. 아키텍처 제안 (모델 레지스트리 + 티어 라우팅)

```
┌───────────────────────────── data_classifier (엔진) ─────────────────────────────┐
│  T1 정규식·deny-list   T2 NER(Presidio)   T3 신경망(교체형)   앙상블   결정론floor │
│                                    │                                                │
│                         NEURAL_BACKENDS 레지스트리                                  │
│   ┌─────────────┬──────────────────┬────────────────────┐                          │
│   │ n2sf-small  │ n2sf-base(현재)  │ n2sf-large         │  ← 같은 교사로 증류된 3종 │
│   └─────────────┴──────────────────┴────────────────────┘                          │
│           ▲ 정적: 프로필로 1개 선택   /   동적: 캐스케이드 승격                      │
└──────────────────────────────────────────────────────────────────────────────────┘
        평가: 하네스가 3종을 동일 held-out으로 비교 → (정확도·지연·크기) 선택 매트릭스
```

- **엔진(3-tier)은 그대로**, 신경망 tier만 크기별 학생으로 교체 → 코드 변경 최소.
- 하네스가 **증류→평가→매트릭스**를 자동화 → 새 후보도 한 줄 등록으로 편입.

## 6. 평가·구축 계획 (하네스 확장)

1. **train_soft를 백본 인자화** — `--base` 로 KoELECTRA-small / mDeBERTa-base / KLUE-RoBERTa-large 선택해 **동일 교사 라벨로 증류**.
2. **3종 학생 생성** → 각각 `models/n2sf-{small,base,large}/`.
3. **동일 held-out으로 일괄 평가** → **선택 매트릭스** 산출:

   | 모델 | F1 | C재현율 | 지연 | 크기 | RAM | 권장 용도 |
   |---|---|---|---|---|---|---|
   | n2sf-small | … | 1.0 | … | … | … | 엣지/실시간 |
   | n2sf-base | 0.80 | 1.0 | 0.6s | 558MB | 1.1GB | 표준 |
   | n2sf-large | ~0.85 | 1.0 | … | ~1.3GB | ~2.2GB | 정확도 |

4. **(옵션) 캐스케이드 라우터** 구현 + 평균 지연/정확도 측정.
5. 산출물: 3 모델 + 선택 가이드 + (선택) 캐스케이드.

## 7. 리스크 / 유의

- **라지 학습 자원**: XLM-R-large/KLUE-large는 fp32 시 MPS OOM 위험(이미 base에서 경험) → 작은 배치·grad accum·양자화 필요. 시간도 2~4배.
- **계열 변경 = 재튜닝**: 백본이 바뀌면 앙상블·임계를 재보정해야 함(base 설정 그대로 못 씀).
- **증류 상한 = 교사**(GPT-4o 0.915 / o4-mini 0.972) — 라지 학생도 이 위로는 못 감.
- **실문서 검증 선행**: 라인업 수치는 합성 held-out 기준 → 실서비스 전 실문서 재검증 필수.
- **소형 티어 안전성**: 작을수록 과대분류/누락 위험 커질 수 있어, **C재현율 1.0 게이트**를 티어별로 강제.

## 8. 권고 요약

1. **KLUE-RoBERTa-large**를 정확도(라지) 티어 1순위로 증류 → 0.85 도전(온디바이스 부담은 XLM-R-large보다 적음).
2. **KoELECTRA-small**를 Fast 티어로 증류 → 엣지/실시간용.
3. 현재 **mDeBERTa-n2sf(base)** 는 Balanced 티어로 유지.
4. 배포는 **정적 프로필 선택** 기본 + 필요 시 **동적 캐스케이드**로 확장.
5. 하네스로 3종 증류·평가 매트릭스를 자동 산출 → **용도별 선택 근거** 확보.

> 진행 중: o4-mini 교사 재증류(base) — 결과가 "라지 없이 base로 얼마나 가능한가"의 근거가 됨.

### 9.1 예측 vs 실측 — 바뀐 점
| 사전 예측(§1~8) | 실측 | 판정 |
|---|---|---|
| KLUE-large가 정확도 1순위(~0.85), XLM-R보다 유리 | **XLM-R-large 0.917 > KLUE 0.872** | ❌ 뒤집힘 — **XLM-R-large가 1순위** |

### 9.2 확정 권고 (실측 기반)
1. **정확도 티어 = XLM-R-large-n2sf (0.917)** — 예측의 KLUE 대신 **XLM-R-large**로 변경. 다국어라 §다국어 적대공격에도 강건.
2. **KLUE-large**는 "한국어 특화·용량 절감(1.3GB)" 대안으로 유지(0.872).
3. **앙상블 비권장** — 이득 없이 자원만 2배.
4. **Fast=small(0.71/13ms/57MB), Balanced=base(현재), Accurate=XLM-R-large** 3-티어 확정.
5. **전 티어 기밀 재현율 1.00**(누락 0) — 안전성 크기 무관 유지.

> 근거 파일: `lineup/matrix.json`, `lineup/report.html`, `distill_soft/llm_baselines.json`.
> base 티어 추가 개선(0.8 확실화)은 `base_0.8돌파_전략.md`의 균형샘플링·온도증류로 진행 중(`n2sf-base-kd`).
