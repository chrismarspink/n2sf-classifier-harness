# 0.80/0.85의 의미 & 라지 모델 검토

> 일자: 2026-07-02 · 대상: mdeberta-n2sf(온디바이스 3-tier) held-out 성능
> 목적: (1) 0.80·0.85 수치의 정확한 의미, (2) 라지 모델로 0.85 기대 가능성과 득실

---

# 1. 수치(0.80 / 0.85)의 정확한 의미

## 1.1 이 숫자는 무엇인가 — **held-out macro-F1**

- 우리가 말하는 "0.80"은 **held-out 평가셋에서의 macro-F1** 값이다.
- **held-out** = 학습에 **쓰지 않은** 72개 문서(GPT-4o/o4-mini/Gemma가 만든 **다양한 문서 분포**). 과적합에 속지 않으려고 학습과 분리한 "정직한 시험지".
- **macro-F1** = 세 등급(O·S·C) 각각의 F1을 낸 뒤 **단순 평균**. (등급 개수가 불균형해도 각 등급을 동등하게 취급)

## 1.2 F1 · precision · recall 정의

| 지표 | 정의 | 의미 |
|---|---|---|
| **정밀도(precision)** | 어떤 등급으로 **예측한 것 중** 실제로 맞은 비율 | "C라고 한 것 중 진짜 C" — 오탐(과대분류) 적을수록↑ |
| **재현율(recall)** | 실제 그 등급 **중** 맞게 잡은 비율 | "진짜 C 중 잡아낸 것" — 누락 적을수록↑ |
| **F1** | 정밀도·재현율의 조화평균 | 둘의 균형 (하나만 높아선 안 됨) |
| **macro-F1** | O·S·C F1의 평균 | 전체 균형 점수 (완벽=1.0) |

## 1.3 우리 0.80 설정의 실제 지표 (held-out 72문서)

| 지표 | 값 | 해석 |
|---|---|---|
| macro-F1 | **0.797** | 세 등급 균형 점수 |
| 정확도(accuracy) | 0.806 | 72문서 중 정확히 맞춘 비율 (~58/72) |
| **기밀(C) 재현율** | **1.00** | **진짜 기밀을 하나도 안 놓침 = 유출 0** |
| 과소분류율(under) | **0.00** | 실제보다 낮게 =유출 방향 오류 **없음** |
| 과대분류율(over) | **0.194** | 실제보다 **높게** 오분류 ~14/72 (안전하나 과잉경보) |

→ **"0.80"의 실체**: **기밀은 100% 잡고(누락 0), 대신 하위 등급 문서의 ~19%를 상위로 과잉분류**해서
정밀도가 깎여 macro-F1이 0.80에 머문다. **안전(누락0)은 확보, 손실은 전부 "과대분류"쪽.**

## 1.4 0.80 → 0.85 는 구체적으로 무슨 차이인가

- 72문서 기준 macro-F1 0.80→0.85 ≈ **과대분류를 ~14건 → ~9건으로 5건가량 줄이는 것**(정밀도 개선).
- 즉 "안 놓치는 건 그대로, **과잉경보를 줄여** 사용성·정밀도를 올리는" 방향.
- **주의(작은 표본)**: 평가가 **72문서**라 각 수치엔 **±0.03~0.05 노이즈**가 있다. 0.80과 0.85 차이(5점)는
  문서 몇 건 차이라, 확정하려면 평가셋을 수백~수천으로 키워야 한다.

## 1.5 기준선(anchor)으로 본 위치

| 성능 | macro-F1 | 비고 |
|---|---|---|
| 무작위 수준 | ~0.33 | 3등급 찍기 |
| 증류 전 시작점 | 0.53 | 규칙+제로샷 |
| **현재(0.80)** | **0.797** | 증류+뉴럴우선 앙상블, 온디바이스 CPU |
| 목표(0.85) | 0.85 | 과대분류 추가 억제 필요 |
| Gemma2:9b(로컬 LLM) | 0.873 | 9B |
| GPT-4o(교사) | 0.915 | 클라우드 |
| o4-mini | 0.972 | 클라우드 |

→ **0.53의 격차를 대부분 메워 LLM(0.92)에 근접**. 남은 격차 ~0.12. **누락 0이라는 안전성은 상용 LLM과 동급.**

## 1.6 반드시 함께 볼 것 (숫자 하나로 판단 금지)

- macro-F1은 요약일 뿐. **보안 맥락에선 "기밀 재현율(누락 0)"이 F1보다 중요**하다 — 현재 모든 설정에서 1.00 유지.
- 이 수치는 **합성/LLM 생성 문서** 기준. **실제 고객 문서 성능은 별도 검증 필요**(현재 미측정).

# 2. 라지 모델로 0.85를 기대할 수 있는가

## 2.1 근거 — 왜 라지 모델이 도움이 되나

- 현재 학생은 **mDeBERTa-v3-base**(BERT 계열 **base 티어**. 백본 ~86M, 다국어 어휘 임베딩 포함 총 279M/558MB).
  "small"이 아니라 **base**이며, "소형"은 **9B LLM 대비** 상대적 표현이었다. held-out에서 **~0.78~0.80에서 상한**.
- 진단상 병목은 규칙/앙상블이 아니라 **학생의 표현 용량**. 증류에서 **더 큰 학생일수록 교사 결정경계를 더 많이 흡수**(지식증류 일반 경향).
- 교사가 GPT-4o(0.915)·o4-mini(0.972)로 충분히 높으므로 **학생 상한을 올릴 여지 = 학생 크기**.

**정정(중요)**: **mDeBERTa-v3-large(다국어판)는 공식 미출시**다(DeBERTa-v3-large는 **영어 전용**).
따라서 한국어/다국어 "large" 후보는 **XLM-RoBERTa-large(~550M)** 또는 InfoXLM-large 등 **다른 계열**이 된다.

→ **기대: XLM-R-large(~550M) 급이면 0.83~0.88 도달 가능성**. **0.85는 현실적 기대치이나 보장은 아님**(0.82~0.87 밴드).
   (계열이 바뀌므로 재증류·튜닝이 필요하고, base에서 쌓은 설정을 그대로 못 쓸 수 있음.)

## 2.2 얻는 것 (Gains)

| 항목 | 내용 |
|---|---|
| **정확도/F1↑** | 0.80 → **~0.85 기대**, 과대분류 감소(정밀도↑) |
| **일반화↑** | 다양·미묘·적대 문서에서 더 안정적 |
| **안전 유지** | 기밀 누락 0 유지 가능(설정으로 보장) |
| **온프레미스 유지** | 여전히 로컬·CPU, 외부 반출 0 (LLM 대비 우위 유지) |

## 2.3 잃는 것 (Costs / Trade-offs)

| 항목 | base(현재) | large(예상) | 영향 |
|---|---|---|---|
| **파라미터** | 279M | ~560M+ | 2배+ |
| **디스크** | 558MB | ~1.1~1.3GB | 2배 |
| **메모리(fp32)** | ~1.1GB | ~2.2~2.5GB | 2배 (양자화 시 ~0.7GB) |
| **추론 지연(CPU)** | ~0.6s/문서 | **~1.5~3s/문서** | 2~4배 느림 |
| **학습 비용** | 중 | 큼(MPS OOM 위험↑) | 재학습 부담↑ |
| **온디바이스 적합성** | 일반 노트북 OK | 저사양·엣지 어려움 | **핵심 가치 훼손** |

## 2.4 전략적 긴장 — "작고 빠른 온디바이스"라는 정체성

- 본 과제의 핵심 포인트는 **"고객 디바이스 자원으로, 외부 LLM·GPU 없이, 빠르게"**.
- 라지 모델은 **크기·메모리·지연을 2~4배** 늘려 이 강점을 잠식 → "그럴 바엔 LLM 쓰지?"라는 반문에 취약.
- **완충책**: ① **양자화(int8/ONNX)** 로 크기·지연 절반, ② **tier early-exit**(정규식 확정 문서는 뉴럴 우회)로 라지 모델이 **소수 애매 문서에만** 작동 → 평균 지연 상승 제한.

## 2.5 라지 외의 대안 (0.85 도달 경로)

| 경로 | 기대 | 비용 |
|---|---|---|
| **더 강한 교사(o4-mini) 재증류** (진행중) | base 상한 소폭↑(~0.80~0.83) | 라벨링 느림, 상한 제한적 |
| **규칙 오탐 정밀화** | 과대분류 일부↓ | 이미 config 스윕은 정체(0.797) |
| **라지 모델 증류** | **0.85 가장 유망** | 크기·지연 2~4배 |
| **하이브리드 확장(escalation)** | 애매 문서만 LLM/라지로 | 일부 문서 지연↑, 구조 복잡 |
| **평가셋 확대(수천)** | 수치 신뢰도↑(성능 자체는 아님) | 생성 비용 |

## 2.6 권고 (판단용)

- **0.85가 "실서비스 필수 기준"이면** → 라지 모델 증류(+양자화+early-exit)가 가장 확실. 크기·지연 증가는 양자화·티어로 완충.
- **"작고 빠름"이 최우선이면** → base 0.80 + 과대분류 억제로 실용화하고, **애매 문서만 라지/LLM로 확장(escalation)** 하는 하이브리드가 균형점.
- 어느 경우든 **실문서 표본 검증**이 선행되어야 0.80/0.85의 실제 의미가 확정된다.

# 3. LLM 지식증류(Knowledge Distillation) 상세

## 3.1 의미 — 무엇인가

**지식증류**는 크고 똑똑한 **교사(teacher) 모델**의 판단 능력을 작은 **학생(student) 모델**에
이전(移轉)하는 기법이다. 학생은 정답 라벨만 보고 배우는 대신, **교사가 "어떻게 판단했는지"**를
모방한다. 결과적으로 **작은 모델이 큰 모델에 근접한 성능**을 낸다.

- 비유: 신입(학생)이 정답지만 외우는 게 아니라, **베테랑(교사)의 판단 근거·확신도**를 보고 배운다.
- 우리 맥락: **클라우드 LLM(GPT-4o)을 교사로, 온프레미스 소형 BERT(학생)가 그 판단을 로컬로 흡수**
  → 추론(실서비스)에는 **LLM이 필요 없다**(외부 반출 0·CPU). "LLM 성능을 로컬로 가져오는" 정공법.

## 3.2 hard-label vs **soft-label** (우리가 쓴 방식)

| 구분 | 교사가 주는 것 | 학생이 배우는 정보 |
|---|---|---|
| hard-label | 최종 등급 1개 (예: "C") | 정답만 |
| **soft-label(우리)** | **확률분포** (예: O=0.1, S=0.3, **C=0.6**) | 정답 + **얼마나 확신하는지·등급 간 유사도** |

soft-label은 "이 문서는 C가 유력하지만 S 성격도 있다" 같은 **교사의 뉘앙스**까지 전달 → 학생이
더 풍부하게, 데이터 효율적으로 배운다. 우리는 GPT-4o가 매긴 O/S/C 확률을 목표로 학생을 학습시켰다.

## 3.3 방법 — 우리가 실제로 한 절차

```
1) 데이터 준비   : 다양한 문서(Azure LLM 생성) 3,873건 — 분포 다양화
2) 교사 라벨링   : GPT-4o가 각 문서에 O/S/C 확률분포 부여
                   예) 기밀문서 → [0.0, 0.0, 1.0], 공지 → [0.9, 0.1, 0.0]
3) 학생 KD 학습  : mDeBERTa(학생)의 출력분포가 교사분포에 가까워지도록 손실 최소화
4) 정직한 평가   : held-out(학습 미사용)으로 일반화 측정
5) 반복(무인 루프): 교사라벨 누적 → 재학습 → 평가 (사이클)
```

**KD 손실(soft cross-entropy / KL):**
```
L = - Σ_g  p_교사(g) · log p_학생(g)      (g ∈ {O, S, C})
```
학생의 log-softmax 분포를 교사 확률에 맞춘다. (고급: temperature T로 분포를 부드럽게,
hard+soft 혼합 α로 정답 라벨도 일부 반영 — 우리는 순수 soft로 시작.)

## 3.4 우리 결과 (증류 효과)

- 학생 뉴럴 단독: **0.53 → 0.78** (교사 GPT-4o 0.915에 근접) — **증류가 실제로 작동**.
- 전체 파이프라인: 앙상블 재튜닝까지 더해 **0.80**.
- **증류의 상한 = 교사 성능**(0.915). 학생은 교사를 못 넘고, **학생 용량이 작으면 교사에 못 미친다**
  → 여기서 "라지 모델" 논의(§2)로 연결된다. 더 강한 교사(o4-mini 0.972)면 상한이 올라간다.

## 3.5 장점 / 한계

| 장점 | 한계 |
|---|---|
| 작은 로컬 모델이 **LLM급 판단 흡수** | **증류 상한 = 교사 성능**(교사 오류도 전파) |
| 추론 시 **LLM·GPU 불필요**(온프레미스·CPU) | **학생 용량 병목** — 너무 작으면 못 따라감 |
| soft-label로 **데이터 효율↑** | 교사 라벨링 **비용/시간**(1회성 오프라인) |
| 실문서를 외부로 안 보냄(교사엔 **합성만** 전송) | 라벨 품질·분포 편향에 민감 |

## 3.6 고급 변형 (향후 옵션)

- **temperature scaling**: 교사 분포를 부드럽게(T>1) → 등급 간 관계 정보 강화.
- **hard+soft 혼합**: `α·soft + (1-α)·정답라벨` → 안정성↑.
- **로짓 매칭 / feature distillation**: 확률뿐 아니라 중간 표현까지 모방 → 더 강한 전이.
- **데이터 증강 증류**: 교사로 대량 다양 문서를 라벨링(무한 라벨) → 학생 일반화↑.
- **라지 학생 증류**: §2 — 학생 용량을 키워 교사에 더 근접(0.85 유망 경로).

## 3.7 왜 이노티움에 적합한가

- **데이터 주권**: 실서비스 추론은 100% 로컬. 교사(클라우드 LLM)는 **개발 단계에서 합성 문서로만** 사용.
- **비용**: 추론당 API 과금 0(로컬 CPU). 교사 비용은 학습 1회성.
- **성능**: "외부 LLM 없이 로컬에서 LLM에 근접"을 실측으로 입증(0.53→0.80, 교사 0.92).

## 부록. 지금 진행 중

- **o4-mini 교사 재증류**가 백그라운드 실행 중(~18시 종료 예정). base 학생으로 0.80을 얼마나 넘는지가
  "라지 모델이 필요한가"의 판단 근거가 된다(base로 0.83+ 나오면 라지 없이도 근접 가능).
