# 1부. 모델의 물리적 실체

## 1.1 모델 =작 "폴더 하나(파일 묶음)"

"모델"은 프로그램이 아니라 **숫자 덩어리(가중치) 파일**이 본체다. mdeberta의 실제 구성:

| 파일 | 크기 | 정체 |
|---|---|---|
| **model.safetensors** | **~558 MB** | **가중치(약 2.8억 개 숫자 행렬)** — 모델의 "두뇌". 본체 |
| config.json | ~1 KB | 아키텍처 명세(층 12·hidden 768·vocab 251k) + **라벨 매핑(id2label)** |
| spm.model / tokenizer.json | ~20 MB | 토크나이저(문장→토큰 변환표) |
| special_tokens_map / tokenizer_config | 작음 | 토크나이저 설정 |

즉 **모델 = 「가중치 + 설정 + 토크나이저」 폴더**.

## 1.2 파인튜닝하면 어떻게 "재생성"되나

```
mDeBERTa 원본 (MoritzLaurer, 라벨 = entailment/neutral/contradiction)
        │  ← 우리 라벨 데이터(C/S/O, 1.8만건)로 추가 학습(가중치 미세 조정)
        ▼
innoMdeBERTa v1 (라벨 = O/S/C, 우리 도메인에 맞게 숫자가 바뀐 새 폴더)
```

물리적으로 일어나는 일:
1. **아키텍처(층·구조)는 그대로** — 558MB 가중치의 **숫자 값들이 우리 데이터 방향으로 미세 조정**.
2. **분류 헤드 교체** — 현재 라벨 `entailment/neutral/contradiction`(NLI용)을 떼고 **`O/S/C` 3-클래스 헤드**를 붙여 학습 → config의 `id2label`이 `{0:공개, 1:민감, 2:기밀}`로 변경.
3. **결과물 = 같은 구조의 새 폴더**(model.safetensors + config + 토크나이저), 크기도 거의 동일(~558MB). **숫자와 라벨만 우리 것**.

동작 방식 변화: 현재는 "기밀이다/아니다"를 3번 묻는 **제로샷(NLI)** → 파인튜닝 후엔 문서를 넣으면 **바로 C/S/O 확률**을 내는 직접 분류기 → **더 빠르고 정확**.

## 1.3 결과물: "innoMdeBERTa" (우리 소유 모델)

```
models/innoMdeBERTa-v1/
├── model.safetensors      ← 우리 도메인 지식이 담긴 가중치 (우리 IP)
├── config.json            ← id2label: {0:공개, 1:민감, 2:기밀}
├── tokenizer.json, spm.model, ...
└── README(model card)     ← 학습 데이터·버전·성능·한계·라이선스
```

- **온전히 우리 소유의 모델 폴더** — 사내 서버에 두고 오프라인 구동, 버전관리(v1→v2…), 재배포 가능.
- 하네스 등록 한 줄: `NEURAL_BACKENDS["inno-mdeberta"] = {"kind":"finetuned","model":"models/innoMdeBERTa-v1"}` → 기존 백엔드처럼 자동 평가·비교.
- **원본을 덮어쓰지 않음** — 원본은 그대로 두고 **파생 모델(새 버전)** 생성. 마음에 안 들면 원본에서 재학습.
- **라이선스**: 원본 mDeBERTa가 MIT → 파인튜닝·상업이용·재배포 허용(고지 유지). 바뀐 가중치(도메인 학습 결과)는 **우리 자산**.
- 경량화 필요 시 **양자화(int8)·증류**로 ~280MB 이하 가능(엣지 배포).

> **비유**: 원본 = "세계 상식을 갖춘 신입". 파인튜닝 = "우리 등급 기준 정답지 1.8만 장으로 OJT". 끝나면 구조·덩치는 같지만 **머릿속(가중치)이 우리 기준으로 재편된 전속 검열관(innoMdeBERTa)**. 그 머릿속 558MB 파일이 복제·배포·버전관리 가능한 IP.

# 2부. 릴리즈(배포) 방식 — 타 과제·업체가 사용하는 법

## 2.1 릴리즈 대상 = 3가지 산출물

| 산출물 | 내용 | 크기 |
|---|---|---|
| 모델 가중치 | `innoMdeBERTa-v1/` | ~558MB |
| 분류 코드 | `data_classifier.py` + `weights.json` | 작음 |
| 런타임 의존성 | presidio·spacy·torch 등 | 큼(GB) |

핵심 난점: 모델·의존성이 무거움 → **git에 가중치 금지(>100MB)**, 배포 형태로 묶어야 함.

## 2.2 배포 형태 (용도별, 추천순)

### ① Docker 이미지 — 외부 업체·타 과제 ★ 가장 권장
코드+모델+의존성을 **하나의 이미지**로 묶음 → 어디서든 `docker run`. **폐쇄망(air-gap) 설치 가능**.
```bash
docker load < innoclassifier-v1.tar          # 인터넷 없이 반입
docker run -p 8080:8080 innoclassifier:1.0
curl -F file=@문서.xlsx localhost:8080/classify   # → {"grade":"C", ...}
```
- 환경 의존성 0(받는 쪽 설치 불필요), 버전 고정, 온프레미스. N²SF 폐쇄망에 tar 반입 가능.

### ② REST API 서비스 — 사내 여러 과제 공유
`classify()`를 FastAPI로 감싸 사내 서버 1곳 배포 → 다른 과제는 **HTTP 호출만**.
- 소비측 ML 의존성 불필요, 중앙 관리·업데이트 용이. (①의 Docker를 사내 서버에 띄우면 동일)

### ③ Python 패키지(wheel) — 파이썬 임베딩 과제
`pip install innoclassifier`(사내 PyPI/Nexus). 모델은 동봉 또는 최초 1회 다운로드.
- 코드 직접 통합. 단, 받는 쪽도 무거운 의존성 설치 필요.

### ④ ONNX 내보내기 — 비(非)파이썬 업체(Java·C#·C++)
모델을 **ONNX 포맷**으로 변환 → ONNX Runtime으로 어느 언어에서나 추론.
- 프레임워크 독립·경량·고속(+양자화). 단, 정규식/NER 전처리는 별도 포팅 필요.

## 2.3 모델 가중치는 어디에 두나 (git 아님)

- **모델 레지스트리/아티팩트 저장소**: 사내 MinIO·Nexus·S3, 또는 Hugging Face 프라이빗 repo.
- **GitHub Release 첨부(assets)** 또는 **Git LFS** — 코드 repo와 분리.
- 코드(GitHub) ↔ 가중치(레지스트리)를 **버전으로 연결**(code v1.0 ↔ model v1).

## 2.4 릴리즈에 반드시 포함

- **버전**: 시맨틱(`1.0.0`) + 모델 버전 분리, CHANGELOG.
- **모델 카드**: 학습 데이터·성능(난이도별 F1)·용도·**한계(L3 위장 취약)**·라이선스.
- **라이선스/고지**: mDeBERTa·presidio·spaCy 모두 MIT(상업·재배포 가능). **third-party 라이선스 고지 동봉**. 외부 업체엔 사용약관·책임범위 명시(오분류 면책·human-in-the-loop 권고).
- **재현 정보**: 어떤 데이터·하네스 버전으로 만든 모델인지.

## 2.5 추천 경로 (이노티움 맥락)

| 받는 쪽 | 권장 형태 |
|---|---|
| 사내 다른 과제 | Docker REST API 1곳 배포 → HTTP 호출 (또는 사내 PyPI wheel) |
| 외부 업체 | Docker 이미지 tar 납품(폐쇄망 반입) + 라이선스 계약 |
| 비파이썬/엣지 | ONNX(+양자화) 모델 + 최소 런타임 |

→ 공통 1순위: **Docker 이미지**(온프레미스·폐쇄망·환경독립 — N²SF에 가장 적합).
