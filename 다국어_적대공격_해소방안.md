# 다국어 적대적 공격 우려 & 해소 방안 (KLUE-RoBERTa-large 선정 시)

> 일자: 2026-07-02 · 맥락: 정확도 티어로 **한국어 특화 KLUE-RoBERTa-large** 채택 시
> **비한국어·언어혼용을 이용한 등급 회피(기밀 누락)** 우려. 방어 설계를 제안한다.

---

## 1. 위협 정의 — 왜 위험한가

**핵심 리스크**: KLUE-RoBERTa는 **한국어 전용 사전학습** → 비한국어/혼용 문서의 의미를 제대로 못 읽어
**기밀을 공개로 오분류(under-classification = 유출)** 할 수 있다. 보안상 **가장 치명적인 오류 방향**.

### 공격 벡터 (등급 회피)
- **언어 전환**: 기밀 내용을 영어/일본어/중국어로 작성 → 한국어 모델이 "의미상 기밀"을 인지 못함.
- **코드 스위칭**: 한 문서에 한국어+외국어 혼용, 핵심 기밀만 외국어로.
- **로마자·음차**: "dae-oe-bi", "gimil" 등 한글을 로마자로 표기 → 키워드·의미 회피.
- **번역 우회**: 원문을 외국어로 바꿔 저장/전송.

> ※ 이 위협은 **KLUE(한국어 전용)** 에 특히 크고, 다국어 백본(mDeBERTa/XLM-R)에는 상대적으로 작다.

---

## 2. 이미 있는 방어 (언어 무관 tier)

3-tier 구조 덕분에 **일부 위협은 이미 차단**된다:

| 방어 | 언어 무관? | 커버 |
|---|---|---|
| **T1 정규식·deny-list** | ✅ | 주민번호·카드(Luhn)·여권·계좌·AWS/API키·이메일·IP — **주변 언어와 무관하게 검출** |
| **다국어 기밀 키워드** | ✅ | `기밀/대외비/극비` + `confidential/secret/機密/社外秘/机密/秘` (이미 등재) |
| 대량 PII·마이넘버 floor | ✅ | 구조적 식별자 다수 → 강제 상향 |

→ **구조적 식별자·명시적 기밀 라벨**이 있으면 **외국어 문서라도 T1/키워드가 잡아** 상향한다.
**잔여 위험 = 구조적 식별자 없이 "의미만 기밀"인 비한국어 문서** (뉴럴 tier만이 판단 가능한 케이스).

---

## 3. 해소 방안 (다층 방어)

### A. 다국어 백본을 "안전 주력"으로 (가장 근본적)
- 보안 분류처럼 **언어 회피가 위협**인 용도에서는 **다국어 백본(mDeBERTa-v3 / XLM-R)** 을 주력으로.
- KLUE(한국어 전용)는 **한국어만 유입되는 폐쇄 코퍼스**에서만 단독 사용.
- 트레이드오프: 한국어 단독 정점 정확도는 다소↓일 수 있으나 **언어 강건성(누락 방지)↑** — 보안상 우선.

### B. 언어 감지 → 라우팅 (Korean/비Korean 분기)
- 문서 언어 감지(경량 langid) → **한국어 = KLUE, 비한국어/혼용 = 다국어 모델**로 라우팅.
- 혼용 문서는 **다국어 모델 우선**(안전측).
- 라인업(소형/base/라지)에 **"언어" 축을 추가**한 선택 규칙.

### C. Korean + 다국어 **앙상블 (상향-only)**
- KLUE(정확) + mDeBERTa(다국어)를 함께 돌려 **더 높은 등급 채택**(escalate).
- 한 모델이 언어 때문에 놓쳐도 **다른 모델이 잡으면 상향** → **누락(under) 억제**. 과대분류는 늘 수 있으나 안전 우선.

### D. 다국어·적대 데이터로 **증류 학습**
- 증류 학습셋에 **영어/일본어/중국어/코드스위칭/로마자 기밀 문서**를 포함(Azure LLM으로 생성, 교사가 라벨).
- 한국어 base 모델도 **다국어 기밀 예시를 학습하면 cross-lingual 강건성↑**.
- 핵심: **"의미만 기밀인 비한국어" 예시**를 충분히 넣어 뉴럴이 배우게 함.

### E. 정규화·음차 처리
- 전각→반각(적용됨), **로마자→한글 음차 정규화**, 유니코드 혼용 정리 → 키워드·정규식 회피 완화.

### F. 다국어 **적대 평가셋(레드팀)** — 하네스로 상시 측정
- held-out에 **언어-적대 난이도(L5)** 추가: 비한국어/혼용/로마자 기밀 문서.
- **핵심 지표 = under_rate(누락)** — 이 셋에서 누락 0 유지되는지 상시 검증.
- 모델 후보별로 이 셋의 누락률을 비교 → **언어 강건성을 정량 선택 기준**으로.

### G. 캐스케이드 승격 (확장)
- 문서가 **비한국어/혼용**이거나 한국어 모델 **신뢰도 낮음** → **다국어 모델/(옵션)LLM으로 승격**.
- 평균 지연은 낮게, 위험 케이스만 강한 모델이 재판정.

---

## 4. 위협 모델별 권고

| 운영 상황 | 권고 |
|---|---|
| **한국어만 유입(폐쇄 코퍼스)** | KLUE-large 단독 OK (+ 언어 감지로 예외 유입 시 경보) |
| **다국어 유입 가능(일반)** | **다국어 백본(mDeBERTa/XLM-R) 주력** + 다국어 적대 증류 + 적대 평가셋 |
| **최고 안전 필요** | **Korean+다국어 앙상블(상향-only)** 또는 캐스케이드 + 언어 무관 T1/키워드 floor |

> **결론**: KLUE-large는 "한국어 정확도"엔 유리하나 **언어 회피 위협이 있으면 단독 주력은 위험**.
> → 다국어 백본 주력화 / 언어 라우팅 / 앙상블 중 택1 + **다국어 적대 학습·평가**를 반드시 병행.

---

## 5. 검증 계획 (하네스)

1. **언어-적대 생성기 추가** — Azure로 EN/JP/CN/혼용/로마자 기밀·공개 문서 생성(교사 라벨).
2. **적대 held-out(L5) 구축** → 후보 모델별 **under_rate(누락) 측정**.
3. 비교 매트릭스에 **"언어 강건성(비한국어 누락률)"** 열 추가 → 선택 기준화.
4. 취약 시 → C(앙상블)·B(라우팅)·D(다국어 증류) 적용 후 재측정.

---

## 6. 즉시 반영할 안전장치 (요약)

- ✅ **언어 무관 T1(구조적 식별자)·다국어 기밀 키워드 floor 유지** — 이미 언어 무관.
- ➕ **다국어 기밀 키워드 사전 확대**(일/중/영 + 로마자 변형).
- ➕ **다국어 적대 데이터로 증류**(뉴럴의 의미 판단 언어 확장).
- ➕ **언어 감지 라우팅 또는 다국어 백본 주력화**.
- ➕ **다국어 적대 평가셋으로 누락률 상시 검증**(레드팀 상설화).
