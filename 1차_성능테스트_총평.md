# N²SF 문서 등급 분류(3-tier) — 1차 성능 테스트 총평

> 일자: 2026-06-30 · 대상: `data_classifier.py` (정규식 + NER + 경량 BERT 3-tier, 외부 LLM·GPU 미사용)
> 대시보드: https://claude.ai/code/artifact/1751dd2c-f6af-4673-9a5f-fa15467fca24
> 저장소: https://github.com/chrismarspink/n2sf-classifier-harness

---

## 한 줄 결론

**개념 입증 성공, 실전 적용 전 1차 관문 통과.**
정상·경계 문서는 GPU·거대 LLM 없이 **LLM 수준**으로 분류함을 정량 입증했고, 동시에 **어디서 무너지는지(적대 난독화)와 해결 경로**까지 명확히 드러낸 의미 있는 1차 테스트.

---

## 1. 입증된 것 (성과)

1. **핵심 가설 성립** — 3-tier가 정상(L1)·경계(L2) 문서에서 **macro-F1 1.00**, 로컬 9B LLM(Gemma 0.89)을 정확도·속도 모두 앞섬. early-exit 시 **평균 45배 빠름, GPU 0**.
2. **보안 1순위 지표 양호** — 정상 난이도에서 **기밀 누락(과소분류) 0**. "놓치지 않는다"는 최우선 요건 충족.
3. **자동 결함 발견·교정** — 사람이 못 본 결함(전화번호↔계좌 정규식 오탐 → 과대분류 44%)을 하네스가 찾아 1.00까지 교정. 평가 인프라가 실제 작동.

## 2. 드러난 한계 (정직하게)

1. **적대 난독화(L3)에서 붕괴** — `대 외 비`(띄어쓰기), 전각 숫자, 가짜 주민번호·미끼카드 앞에서 **F1 0.54 정체, 가끔 기밀 누락**. 72회 반복해도 못 넘음. → 정규식 회피가 근본 취약점.
2. **합성 데이터 기반** — 만점은 "이 시험에서 완전 분리 가능"이지 실문서 성능 보장이 아님. 실문서 검증 미실시.
3. **뉴럴은 제로샷(미학습)** — 도메인 학습이 없어 위장에 약함.

## 3. 1차 테스트의 진짜 의의

점수(1.00/0.54)보다 **검증 체계를 갖췄다는 것**이 더 큰 성과:
- 난이도별로 **모델의 실력 곡선과 한계선을 수치로 그림** (L1·L2 통과 / L3 벽).
- BERT 12종 중 **mdeberta 최적**임을 데이터로 확정.
- 이후 어떤 개선이든 **같은 잣대로 재측정** 가능 → 개선 효과를 객관적으로 증명할 토대 확보.

## 4. 핵심 수치

| 구분 | 결과 |
|---|---|
| 기본 설정 → 자동 최적화 (정상·경계) | macro-F1 0.47 → **1.00**, 과대분류 0.44 → 0.00 |
| 최적 BERT vs Gemma2:9b (정상) | **1.00 / 0.89**, 속도 ~23배(early-exit 45배) |
| 적대 난독화(L3) | **F1 ~0.54 (정체)** |
| 최적 모델 | **mdeberta** (mDeBERTa-v3 다국어 제로샷, 279M·552MB·CPU ~0.6s) |
| 주말 자동 최적화 | 84회 반복, L1→L3, 18,096문서/162,864파일 |

## 4-1. Gemma2:9b(로컬 LLM) 대비 성능 ★

동일 텍스트를 같은 정책 기준으로 분류해 비교(test 162파일, 정상 난이도 기준).

| 항목 | **최적 BERT 3-tier (내 모델)** | Gemma2:9b (로컬 LLM) | 우위 |
|---|---|---|---|
| macro-F1 (정확도) | **1.00** | 0.89 | 내 모델 우세 |
| 기밀(C) 재현율 | 1.00 | 1.00 | 동일(누락 없음) |
| 분류 지연 p50 | **~812 ms** (early-exit 평균 ~424 ms) | ~18,900 ms | **약 23~45배 빠름** |
| 모델 크기 | 279M / 552 MB | 9B / 5.4 GB | **약 32배 작음** |
| 메모리 | ~1.1 GB (양자화 ~0.3 GB) | ~6~9 GB | 내 모델 경량 |
| 구동 환경 | CPU·일반 노트북 | 고사양·GPU 권장 | 내 모델 우위 |

**요지**: 정상·경계 문서에서 내 3-tier 모델은 **9B 로컬 LLM보다 더 정확하고, 23~45배 빠르며, 32배 작고, GPU도 불필요**.
대표 사례 — Gemma는 "기밀이 아니다" 같은 **부정문맥을 S로 과대분류**(9B도 함정에 빠짐)한 반면, 최적 3-tier(mdeberta+vote)는 정답 처리.

> 단, 위 비교는 정상 난이도(L1) 기준. 적대 난독화(L3)에서는 두 모델 모두 어려움을 겪으며, 이 구간 개선은 2차 과제.

| 정상 문서 분류 | ★★★★★ (LLM 수준·초고속) |

| 적대/위장 문서 | ★★☆☆☆ (개선 필요) |
| 속도·경량성 | ★★★★★ (CPU·로컬) |

## 6. 다음 단계 (우선순위)

1. **전처리 정규화 + 체크섬** — 빠르고 확실. L3 상당수가 이것만으로 회복 가능성 높음(투입 대비 효과 최대).
2. **mdeberta 파인튜닝** — 위장 패턴을 학습으로 일반화. 데이터(L3 1.7만건) 이미 확보.
3. **실문서 표본 확보 → 동일 하네스로 재검증** — 합성↔실전 격차 측정·재보정.

## 7. 상용 LLM(Claude·Gemini) 대비 기대 성능

> 이번 비교군은 로컬 Gemma2:9b 였다. 최상위 클라우드 LLM(Claude Opus·Gemini Pro)과의 관계를 정리한다.

**정확도 관점 (예상):**
- **정상·경계 문서(L1·L2)**: 우리 모델이 이미 ~1.00 → 클라우드 LLM과 **사실상 동급**(둘 다 거의 만점).
- **적대 난독화(L3)**: 클라우드 최상위 LLM은 추론력이 강해 `대 외 비`·가짜 식별자 같은 위장을 **우리(현재 0.54)보다 잘 잡을 가능성 높음**(추정 0.8~0.95). → **현재는 격차 존재**, 파인튜닝·전처리로 좁히는 것이 목표.

**그러나 정확도만으로 결정되지 않는다 — N²SF의 결정적 제약:**

| 항목 | 우리 모델(온프레미스) | 클라우드 LLM(Claude·Gemini) |
|---|---|---|
| **기밀문서 외부 반출** | 불필요(로컬 처리) | **API로 외부 전송 → N²SF·공공·금융 규정상 사용 불가/제약** |
| 데이터 주권 | 완전 통제(망분리 가능) | 사업자 서버 경유 |
| 건당 비용 | 사실상 0(CPU) | 토큰당 과금(대량 처리 시 부담) |
| 속도 | CPU 수ms~수백ms | 네트워크 왕복 포함 |
| 가동 환경 | 폐쇄망·온프레미스 OK | 인터넷 연결 필수 |

> **핵심**: 기밀 등급 분류는 그 대상이 **기밀문서 자체**다. 이를 외부 클라우드로 보내는 순간 보안 정책 위반이 되므로, **최상위 클라우드 LLM은 정확도가 높아도 본 용도에선 채택이 어렵다.** 따라서 현실적 경쟁 구도는 "클라우드 LLM vs 우리"가 아니라 **"온프레미스에서 무엇이 최선인가"**이며, 그 안에서 우리 3-tier는 강력한 후보다. 목표는 **온프레미스 제약을 지키면서 클라우드 LLM 수준 정확도에 근접**하는 것.

## 8. 파인튜닝을 통한 이노티움 전용 등급분류 AI 가능성

**결론: 가능성 높음. 이미 핵심 자산을 보유.**

- **학습 데이터 파이프라인 확보** — 하네스가 등급·난이도·포맷별 라벨 데이터를 자동 생성(주말에 1.8만 문서, 적대 케이스 1.7만 확보). 보통 파인튜닝의 최대 비용인 "라벨 데이터"가 이미 해결.
- **방법** — mdeberta(또는 한국어 인코더)에 3-클래스 분류 헤드를 붙여 **이노티움 정책 + 실제 고객 문서**로 학습 → 위장 패턴·조직 고유 기밀어를 모델이 일반화. 오프라인 1회 학습, 추론은 여전히 CPU 소형 모델(온프레미스 유지).
- **차별화(IP)**:
  - 고객사별 맞춤(사내 프로젝트명·기밀 라벨·문서 양식) 반영 → **범용 LLM이 못 주는 도메인 특화**.
  - 자동 평가·난이도 에스컬레이션 **하네스 자체가 경쟁사 모방 어려운 자산**.
  - "외부 반출 0 + 도메인 특화 + 지속 개선 루프" = 클라우드 LLM·타 벤더 대비 명확한 포지셔닝.
- **현실성** — 소형 모델·중간 연산으로 수주 내 PoC 가능. 점진적 재학습으로 성능 누적.

→ **"이노티움 온프레미스 N²SF 등급분류 AI"** 로 제품화 가능. 클라우드 LLM을 못 쓰는 공공·금융·국방 시장이 타깃.

## 9. 상업적 고려 사항

**강점 (Sell points)**
- **데이터 주권·규제 적합** — 온프레미스·망분리, 외부 전송 0 → N²SF·공공·금융 규정 충족(클라우드 LLM의 구조적 약점을 정면 공략).
- **TCO 우위** — 건당 API 과금 없음, CPU 추론. 대량·상시 처리일수록 클라우드 대비 비용 격차 확대.
- **라이선스** — mDeBERTa(MIT)·Presidio(MIT)·spaCy(MIT) 등 상업 이용 가능(로열티 없음). *배포 전 각 구성요소 라이선스 최종 확인 권장.*
- **설명가능성** — 규칙·SHAP 근거 제공 → 감사·이의제기 대응 용이(블랙박스 LLM 대비 강점).

**고려·리스크**
- **오분류 책임** — 과소분류(기밀 누락)는 유출 위험. 경계 사례 **사람 검토(human-in-the-loop)**, 보수적 임계값, SLA 전 실문서 검증 필수.
- **실데이터 검증 선행** — 현재는 합성셋 결과. 상용화 전 실문서로 정확도·재보정 필요(과대광고 금지).
- **운영(MLOps)** — 패턴·정책 변화에 따른 재학습·업데이트 체계, 버전관리, 모니터링 필요.
- **L3 한계 해소가 상용화 전제** — 위장 문서 대응(전처리·체크섬·파인튜닝) 완료 후 시장 신뢰 확보.

**제품화 형태(안)**
- 온프레미스 어플라이언스/서비스. 2단 구성(규칙 고속 tier + 뉴럴 정밀 tier, early-exit).
- 차등 제공: 기본(규칙+NER, 초고속) ↔ 고정밀(파인튜닝 뉴럴 포함).

## 결론

1차 테스트로서 **합격**. "정상 문서는 거대 AI 없이 충분, 위장 문서는 보완 필요"라는 결론과, 그 보완 효과를 증명할 수 있는 **측정 체계**를 동시에 확보했다. 2차는 ①전처리·②체크섬으로 L3를 얼마나 끌어올리는지가 관건이다.
