# 규칙·정책 확장 방안 — 정규식/키워드 추가 & 회사 규정 참조

> 일자: 2026-07-02 · 질문: (1) 정규식·키워드를 추가해 인식율을 올릴 수 있나?
> (2) 회사 규정/정책을 스크립트로 추가해 분류가 참조하게 할 수 있나? → **둘 다 가능**(설계상 지원).

---

## 1. 정규식·키워드 추가로 인식율 향상 — **가능**

`data_classifier.py`는 규칙 tier(T1)를 **추가만 하면 즉시 인식**하도록 만들어져 있다. 학습 불필요·즉효·언어 무관.

### 1.1 확장 지점

| 대상 | 위치 | 추가 방법 | 효과 |
|---|---|---|---|
| **새 정규식(PII 등)** | `PATTERN_RECOGNIZERS` | dict 1개 추가 → Presidio 인식기로 즉시 등록 | 새 식별자 검출(재현율↑) |
| **고정 어휘(사내명/VIP)** | `DENY_LIST_RECOGNIZERS` | 리스트에 용어 추가 | 프로젝트명·부서명·인물 검출 |
| **등급 키워드** | `GRADE_KEYWORDS` (코드) 또는 `weights["keyword"]` (런타임) | (키워드, 가중치, 라벨) 추가 | 기밀/대외비류 라벨 검출 |
| **엔티티 점수·표시명** | `ENTITY_WEIGHTS` / `ENTITY_LABELS` | 타입별 가중치·라벨 | 새 타입의 등급 기여도 |
| **대량 PII 집계 대상** | `BULK_PII_TYPES` | 타입 추가 | 명부 escalation 반영 |
| **문맥 부스트** | 각 패턴의 `context` | 주변 단어 지정 | 문맥 있으면 점수↑(정밀도↑) |

### 1.2 예시

**새 정규식(건강보험번호):**
```python
PATTERN_RECOGNIZERS.append({
  "name": "KR_HEALTH_INS", "supported_entity": "KR_HEALTH_INS",
  "patterns": [{"name": "hi", "regex": r"\b\d-\d{10}\b", "score": 0.6}],
  "context": ["건강보험", "보험증", "요양기관"]})       # 문맥 있으면 부스트
ENTITY_WEIGHTS["KR_HEALTH_INS"] = 3.0
BULK_PII_TYPES.add("KR_HEALTH_INS")
```

**사내 기밀 어휘(코드 없이 런타임):**
```python
weights = {"keyword": [
   {"keyword": "사외유출금지", "weight": 3.0, "label": "사외유출금지"},
   {"keyword": "Project Nova", "weight": 2.0, "label": "Project Nova"}]}
classify("문서.docx", weights=weights)     # 재배포 없이 즉시 반영
```

### 1.3 인식율에 미치는 영향 & 주의

- **재현율↑(놓침↓)**: 새 정규식/키워드는 **구조적 식별자·명시 라벨**을 더 잡음 → 특히 **기밀 누락 방지**에 직접 기여(가장 값싼 안전 강화).
- ⚠️ **과대분류(오탐) 위험**: 너무 느슨하면 오탐(예: 전화번호↔계좌). 대응:
  - `context` 요구, **체크섬 검증**(주민번호/카드), **supersede**(겹치는 일반 인식기 억제), 가중치 조정.
  - **추가 후 하네스로 재평가**(held-out) → 재현율↑ vs 정밀도↓ trade-off 수치 확인. SHAP로 어떤 규칙이 기여했는지 확인.
- **런타임 vs 코드**: 키워드·가중치·임계는 `weights` 로 무재배포 조정, 새 정규식은 코드/프로필에 추가.

## 2. 회사 규정·정책 참조 — **가능** (정책 외부화)

회사별 **정책을 스크립트(설정 파일)로 주입**해 분류가 참조하게 할 수 있다. 엔진은 그대로, 정책만 교체.

### 2.1 GradeProfile — 등급 체계·정책 외부화 (설계 명세 보유)

`등급프로필_외부화_명세.md`의 구조로, **회사별 프로필(YAML/JSON)**에 다음을 담아 런타임 로드:

```yaml
# profiles/companyA.yaml
name: companyA
grades: [PUBLIC, INTERNAL, RESTRICTED, SECRET]        # 회사 등급 체계
codes:    {PUBLIC: P, INTERNAL: I, RESTRICTED: R, SECRET: S}
thresholds: {SECRET: 6.0, RESTRICTED: 3.0, INTERNAL: 0.75, PUBLIC: 0.0}
keywords:                                              # 사내 기밀 어휘
  - ["사외비", 3.0, "사외비"]
  - ["이사회 자료", 2.0, "이사회"]
deny_list:                                             # 사내 프로젝트/부서/인물
  - ProjectNova
  - 전략기획실
floor_rules:                                           # 결정론적 정책 규칙(아래 2.2)
  - {when: keyword, keyword: "사외비", min_grade: RESTRICTED}
  - {when: entity, entity: KR_RRN, min_grade: SECRET}
regulations: {ko: [KR-PIPA, 사내정보보안지침-제7조]}
```

→ **회사마다 다른 규정을 프로필로** — 같은 엔진이 A사(4등급)·B사(3등급)를 각자 정책대로 분류.

### 2.2 floor_rules — 회사 규정을 "결정론적 규칙"으로 코드화

정책을 **강제 상향 규칙**으로 표현(확률 tier가 못 내림 = 규정 우선):

| 규정 예 | 규칙 표현 |
|---|---|
| "주민번호 포함 문서는 무조건 최고등급" | `{when: entity, entity: KR_RRN, min_grade: SECRET}` |
| "계약·금액 포함 시 최소 대외비" | `{when: keywords_all, any: [계약, 금액], min_grade: RESTRICTED}` |
| "전략기획실 프로젝트명 → 기밀" | `{when: deny, entity: INTERNAL_PROJECT, min_grade: SECRET}` |
| "개인정보 10건 이상 명부 → 기밀" | `{when: bulk_pii, threshold: 10, min_grade: SECRET}` |

- **불변식**: floor는 등급을 **올리기만** 함 → 규정이 요구하는 최저 등급을 **항상 보장**(안전).
- 규정 변경 시 **프로필만 수정**(재학습·재배포 불필요).

### 2.3 그 외 정책 참조 수단

- **regulations 매핑**: 검출 시 규제·사내지침 조항 코드를 결과(`compliance.violations`)에 표기 → 감사·근거.
- **런타임 weights override**: 부서·문서종류별 임계/가중치 즉시 조정.
- **문서 유형별 프로필**: 계약서/인사/기술문서 등 유형별 정책 분리 가능.

---

## 3. 함께 쓰는 법 (권장 운영)

```
회사 규정 문서 → 정책 프로필(YAML)로 번역
   ├─ 정규식/deny-list : 사내 식별자·프로젝트·부서 (인식율↑)
   ├─ keywords         : 사내 기밀 라벨 (재현율↑)
   ├─ floor_rules      : "무조건 이 등급" 규정 (안전 보장)
   ├─ thresholds       : 민감도 정책
   └─ regulations      : 조항 매핑(감사 근거)
        │
        ▼  data_classifier(엔진, 불변) 이 프로필을 참조해 분류
        ▼  하네스로 재평가 → 인식율(재현율)↑ vs 과대분류 trade-off 수치화 → 임계 재보정
        ▼  SHAP로 "어떤 규칙/키워드가 이 등급을 만들었는지" 근거 제시
```

---

## 4. 주의사항

- **과대분류 관리**: 규칙을 늘리면 오탐도 는다 → context·체크섬·supersede·임계로 억제, **추가 시마다 하네스 재평가**.
- **재보정**: 정책 변경 후 held-out 재평가로 등급 분포·누락률 확인(회귀 방지).
- **에어갭**: 프로필·모델 모두 로컬 파일 로드(런타임 외부 접근 금지).
- **버전관리**: 정책 프로필을 형상관리(회사·시점별) → 재현·감사 가능.
- **설명가능성**: 규칙 기반이라 SHAP·근거 제시가 쉬움(LLM 블랙박스 대비 강점).

---

## 5. 결론

1. **정규식/키워드 추가 = 가능·즉효** — `PATTERN_RECOGNIZERS`/`GRADE_KEYWORDS`/`weights` 로 인식율(재현율)↑. 오탐은 context·체크섬·supersede·하네스 재평가로 관리.
2. **회사 규정 참조 = 가능** — **GradeProfile(정책 프로필 YAML)** + **floor_rules(강제 규칙)** 로 회사별 정책을 스크립트화해 분류가 참조. 엔진은 불변, 정책만 교체.
3. 둘 다 **온프레미스·설명가능·재배포 최소**로 동작 → 고객사별 맞춤(제품 차별화 포인트).

> 구현 상태: 정규식/키워드/weights는 **현재 즉시 사용 가능**. GradeProfile·floor_rules 전면 외부화는
> `등급프로필_외부화_명세.md`의 설계대로 구현 필요(유인 작업 권장).
