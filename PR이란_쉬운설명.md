# PR(Pull Request)이란? — 쉬운 설명

> 일자: 2026-07-02 · 질문: "PR이 뭐지?" (방금 GitHub 업로드 맥락)

## 1. 한 문장

**PR(Pull Request) = "이 변경사항을 main(정식 버전)에 합쳐 주세요"라는 병합 요청**이다.
바로 정식 버전에 덮어쓰지 않고, **검토 후 합치는 안전장치**.

## 2. 비유

- `main` 브랜치 = **정식 출판된 책**.
- 피처 브랜치(`update/lineup-distill-deliverables`) = **내가 고친 원고 사본**.
- **PR** = "원고 사본의 이 수정들을 정식 책에 반영해도 될까요?" 하고 올리는 **검토 요청서**.
- 확인·승인(**Merge**)하면 → 정식 책(main)에 반영됨.

## 3. 왜 이렇게 하나 (바로 main에 안 넣고)

- **리뷰 기회**: 합치기 전에 무엇이 바뀌는지 한눈에 보고 점검.
- **안전**: 문제가 있으면 병합 전에 막을 수 있음(정식 버전 보호).
- 그래서 자동 정책이 **main 직접 푸시를 막고** 피처 브랜치+PR을 권장한 것.

## 4. 지금 우리 상태

- 변경사항(108개 파일)은 이미 GitHub의 **피처 브랜치**에 올라가 있음(안전하게 보관됨).
- 아직 **main에는 반영 안 됨** → main에 넣으려면 **PR을 만들고 Merge(병합)** 하면 됨.
- PR 만들기 링크:
  `https://github.com/chrismarspink/n2sf-classifier-harness/pull/new/update/lineup-distill-deliverables`

## 5. 선택지

| 방법 | 설명 |
|---|---|
| **PR 만들고 병합** | 위 링크 → "Create pull request" → "Merge" 클릭. main에 반영 |
| **그냥 두기** | 피처 브랜치에만 보관(백업). 필요할 때 병합 |
| **혼자 쓰는 저장소면** | 리뷰가 굳이 필요 없으면 병합만 바로 눌러도 됨 |

> 혼자 관리하는 저장소이므로, 검토가 필요 없으면 링크에서 **Create → Merge** 두 번 누르면 끝납니다.
> 원하시면 제가 PR 생성·병합을 대신 진행할 수 있습니다(권한 확인 필요할 수 있음).
